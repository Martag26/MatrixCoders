Introducción a los sistemas de control de versiones

¿Qué es el control de versiones?

Un sistema de control de versiones (VCS) registra los cambios realizados en archivos a lo largo del tiempo, permitiendo recuperar versiones anteriores, comparar cambios y colaborar en equipo sin pisarse el trabajo.

Sin control de versiones — el problema

proyecto_final.zip proyecto_final_v2.zip proyecto_final_ESTE.zip proyecto_final_BUENO_de_verdad.zip proyecto_final_entrega.zip ← ¿cuál era el bueno?

Tipos de VCS

TipoEjemploCómo funciona
LocalRCSHistorial solo en tu máquina
CentralizadoSVN, CVSUn servidor central; todos se conectan a él
DistribuidoGit, MercurialCada clon es un repositorio completo

¿Por qué Git es el estándar?

  • Distribuido: trabajas offline con historial completo
  • Rápido: operaciones locales (no dependen de red)
  • Ramas baratas: crear una rama cuesta milisegundos
  • Integridad: cada commit tiene un SHA-1 único; imposible alterar sin rastro
  • Ecosistema: GitHub, GitLab, Bitbucket, CI/CD, GitHub Actions…

Los tres estados de Git

Working Directory → Staging Area (Index) → Repository (.git) (modificado) git add git commit
  • Working Directory: los archivos que editas en tu disco
  • Staging Area: los cambios marcados para el próximo commit
  • Repository: historial permanente de commits
💡 Git no guarda snapshots de diferencias (como SVN), sino snapshots del estado completo del proyecto en cada commit, lo que lo hace extremadamente eficiente.
