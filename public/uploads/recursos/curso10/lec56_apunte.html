Trabajo avanzado con ramas y resolución de conflictos

¿Qué es un conflicto de merge?

Un conflicto ocurre cuando dos ramas modifican las mismas líneas del mismo archivo. Git no sabe qué versión conservar y pide que lo decidas manualmente.

Ejemplo de conflicto

<<<<<<< HEAD (rama actual: main)
console.log("Versión de main");
=======
console.log("Versión de feature");
>>>>>>> feature/login

Para resolver: edita el archivo, elimina los marcadores y quédate con el código correcto. Luego:

git add archivo-conflicto.js
git commit    # Git sugiere mensaje de merge automáticamente

Rebase — historial lineal

# En lugar de merge (que crea commit de fusión)...
git checkout feature/login
git rebase main

# Rebase interactivo — reescribir historial
git rebase -i HEAD~3    # editar últimos 3 commits
Antes del rebase: main: A─B─C \ feature: D─E Después del rebase: main: A─B─C─D'─E' (historial limpio)

git stash — guardar cambios temporalmente

# Guardar trabajo en progreso sin commitear
git stash

# Ver stashes guardados
git stash list

# Recuperar el último stash
git stash pop

# Recuperar un stash concreto
git stash apply stash@{2}

Deshacer cambios

# Deshacer staging (sin perder cambios)
git restore --staged archivo.js

# Descartar cambios en working directory
git restore archivo.js

# Revertir un commit (crea nuevo commit que deshace)
git revert abc1234

# Reset (MUY destructivo — solo en local)
git reset --soft HEAD~1   # deshace commit, conserva staging
git reset --hard HEAD~1   # deshace commit Y cambios
⚠️ Nunca hagas git rebase ni git reset --hard sobre commits que ya subiste al repositorio remoto compartido — reescribirás el historial y causarás problemas a tus compañeros.
