Git y GitHub

Flujos de trabajo paralelos y buenas prácticas

Git Flow — el flujo más completo

main ──────────────────────────── (producción) \ / hotfix fix───────────────────── \ \ release ───────────────────────── \ / develop ──────────────────────── / \ / \ / feature A B── C─── (feature branches)
RamaPropósito
mainCódigo en producción — siempre estable
developIntegración de features — base de desarrollo
feature/*Nuevas funcionalidades
release/*Preparar una versión para producción
hotfix/*Parches urgentes directamente sobre main

GitHub Flow — el más sencillo

  1. Crea una rama desde main
  2. Desarrolla y commitea en esa rama
  3. Abre un Pull Request
  4. El equipo revisa el código
  5. Se fusiona a main y se despliega

Buenas prácticas de commits

# Conventional Commits — formato estándar
feat: add user authentication
fix: correct password validation bug
docs: update README installation steps
refactor: extract login logic to service
test: add unit tests for auth module
chore: update dependencies

Pull Request (PR) — revisión de código

  • Abre un PR descriptivo: qué hace, por qué y cómo probarlo
  • Pide revisión a al menos un compañero
  • Usa las sugerencias de código de GitHub para proponer cambios concretos
  • Pasa los checks de CI antes de fusionar
  • Borra la rama después de fusionar
💡 Configura Branch Protection Rules en GitHub para que main requiera PR aprobado y CI en verde antes de poder fusionar.
